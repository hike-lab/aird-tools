## 0. 로드맵의 성격

본 로드맵은 AIRD 표준 시리즈를 **구성하는 모든 문서의 역할·관계·현황을 하나의 지도로 제시**하는 공식 문서다. 두 가지 질문에 답한다.

1. **지금 어디에 있는가?** 본 표준 초안 4종(STD-001, STD-002, STD-003, STD-005)은 이미 상호 참조 관계까지 형성된 상태로 v0.1이 배포되었다. 이를 정확히 기술한다.
2. **무엇이 비어 있는가?** 본 표준 초안이 배포되었음에도 운영·근거 지원 문서(OPG-001, TR-001)와 적합성 인증 표준(STD-004)은 미작성 상태다.  
 
본 로드맵은 "작성 순서 권고"가 아니라 **"현황 지도 + 공백 식별"** 로서 기능한다.

> **버전 표기 원칙:** 본 로드맵은 각 표준의 **공식 배포 버전인 v0.1**을 기준으로 기술한다. v0.1 배포 시점까지의 개발 과정에서 존재했던 중간 판본은 각 표준 본문의 개정 이력 테이블에서 확인한다.

## 3. v0.1 시리즈의 설계 철학과 핵심 결정사항

본 표준 v0.1 시리즈는 공통의 설계 철학을 공유하며, 이는 STD-001에서 명시적으로 선언된 뒤 나머지 표준에 반영된다.

### 3.1 설계 철학 4대 원칙

- **상태 표준(State Standard)** — 데이터의 상태(Raw / Quality-Ready / Purpose-Ready / Stale)를 명시적으로 정의하고, 상태 전이의 조건·책임·기록 방법을 규정한다.
- **모델-데이터 분리(Model-Data Separation)** — 데이터 자체와 모델 의존 메타데이터를 분리하여, 모델이 교체되어도 데이터 자산이 훼손되지 않도록 한다.
- **위임(Delegation)** — 운영 세부 규칙은 OPG-001에, 가중치 수치 근거는 TR-001에, 탐색·배포는 STD-005에 위임한다. 본 표준은 원칙과 공식을 규정하고, 운영·근거·유통은 별도 문서에 맡긴다.
- **기술스택 중립(Technology Neutrality)** — 특정 구현 기술(A2A, MCP, 특정 벤더 플랫폼 등)을 규범적으로 강제하지 않는다. 추상 인터페이스 요건만 정의하고 구체적 구현은 Informative 부속서로 처리한다.

### 3.2 STD-001 v0.1의 주요 결정

- **6개 품질 차원과 20개 지표** — 측정 공식, 전제 조건, 데이터 유형별 적용 매트릭스
- **Q-Tier 등급 체계** (Q-Bronze / Q-Silver / Q-Gold)와 Gate 임계치
- **DM(진단 성숙도) 3단계 체계** — Q-Tier(품질 수준)와 독립적인 측정 신뢰 수준 축
- **PRECONDITION_UNMET 선언 및 3회 누적 규칙** — 전제 조건 미충족 시 남용 방지 장치
- **신규 용어 7개** — 모델-데이터 결합도, Pack 완전성, Stale 상태, 소비자 피드백 루프, 파생 데이터, 참조 프로파일 URI, Fine-tuning Pack
- **데이터 유형 5종** — STRUCT, TEXT, IMAGE, TSERIES, **FT(Fine-Tuning)**
- **purposeReadinessVector 체계** — ML, DL, RAG, KG, Stats에 더해 **FineTuning, Multimodal, Streaming, Evaluation** 네 항목 추가
- **JSON 리포트 스키마** — Stage 2 산출물로 표준화

### 3.3 STD-002 v0.1의 주요 결정

- **3계층 메타데이터 아키텍처** — Layer 1 (Discovery) / Layer 2 (Understanding) / Layer 3 (Operability)
- **Purpose-Type별 멀티 프로파일** — ML, DL, RAG, KG, Stats 프로파일
- **Stale 상태**를 4번째 상태로 추가 및 자동 전이 조건 정의
- **Layer 3 공통 필수 속성 7개** — packGeneratedAt, sourceDataVersion, modelDataCoupling, packCompleteness, agentAccessible, transformationProfile, multiPurposeLinks
- **Fine-tuning Profile (aird-ft)** 초안 — Extension Profile 등록 절차 진행 예정
- **RAG Profile 필수 속성 4개 추가** — embeddingModelRegistry, embeddingModelCommit, embeddingModelLicense, modelCardURI
- **표준 어휘 우선 적용 원칙** — 커스텀 어휘 도입 기준 명시

### 3.4 STD-003 v0.1의 주요 결정

- **ETL 파이프라인 Stage 구조** — 각 단계의 입출력 아티팩트 명세
- **Gate 판정 체계** — Quality-Ready 전이 통제의 실질적 지점
- **Provenance 기록 규격** — 변환 과정의 추적 가능성 보장
- **Extension Profile 등록 거버넌스** — 신규 Purpose-Type 수용 절차
- **단일 출처 원칙 준수** — Quality-Ready 전이 조건은 STD-002 7.2절을 단일 출처로 인정

### 3.5 STD-005 v0.1의 주요 결정

- **Pack 물리적 교환 규격** — 디렉터리 구조, 압축 방식, 번들링 규칙
- **Pack 레지스트리 구조** — 탐색 인터페이스 및 필터 조건
- **에이전트 소비 프로토콜** — 기술스택 중립 원칙 하의 추상 인터페이스
- **소비자 피드백 루프 API** — 사용 후 피드백의 구조화된 수집
- **계보 연속성 규칙** — 교환 이후 파생 계보가 원본 계보와 연결되도록 보장
- **STD-004(적합성 인증) 참조** — certificationLevel 필드를 탐색 필터로 예약

### 4.1 본 표준 간 상호 참조 매트릭스 (✓ = 실제 본문에 참조 관계 존재)

| **STD-002** | ✓ 측정 지표의 기록 스키마로 DQV 채택 | — | ✓ Layer 3을 Pack 메타데이터로 | — | ✓ Discovery 메타데이터를 탐색 API에 노출 |
| **STD-003** | ✓ Gate 2 판정에 Q-Tier 사용 | ✓ 단일 출처 원칙: Quality-Ready 전이 조건은 STD-002 7.2절 | — | — | ✓ Pack·Provenance 구조 제공 |
| **STD-004** | (예정) | (예정) | (예정) | — | (예정) |

### 4.2 주요 상호 참조 원칙

- **단일 출처 원칙 (Single Source of Truth)** — 어떤 값·조건이 여러 표준에 나타날 때 단일 출처를 지정한다. 예: Quality-Ready 전이 조건 수치는 **STD-002 7.2절이 단일 출처**이며, STD-003은 이를 재정의하지 않고 적용한다.
- **공동 설계 영역** — purposeReadinessVector 및 Layer 3 공통 속성은 STD-001과 STD-002가 **공동 설계**되었다. 한쪽만 고치면 정합성이 깨진다.
- **계보 연속성** — STD-003이 생성한 `provenance.jsonld`는 STD-005 교환 단계에서 단절 없이 이어져야 한다.

### 4.3 본 표준이 지원 문서에 제공하는 것

| 제공 항목 | 수신 문서 | 내용 |
|---|---|---|

| 가중치 고정값 | TR-001 | 수치 근거 작성의 출발점 |

| Pack 구조 | STD-004 | 인증 검증 대상 명세의 기반 |

### 4.4 지원 문서가 본 표준에 공급해야 할 것 (현재 공백)

| 공급처 | 수신처 | 공급 내용 | 공백의 영향 |
|---|---|---|---|
| **TR-001** | STD-001 | 가중치 수치(D3=0.25 등)의 정당화 | 가중치 이의 제기 방어 근거 부재 |
| **OPG-001** | STD-001 | Q-Tier 판정 결과의 유효기간 | Stale 전이 시점 확정 불가 |
| **OPG-001** | STD-001 | PRECONDITION_UNMET 누적 3회 초기화 조건 | 남용 방지 규칙의 리셋 기준 미정 |
| **OPG-001** | STD-001/003 | 진단 도구 인증 기준 | 어떤 도구의 측정 결과를 받아들일지 미정 |
| **STD-004** | STD-005 | 인증 수준(Level A/B/C) 정의 | certificationLevel 탐색 필터 연동 보류 |

> **해석:** 본 표준 시리즈는 기술적으로는 v0.1 공식 배포를 마쳤지만, 위 공급이 이루어지기 전까지는 **운영 단계(Operational Phase)로 진입할 수 없다**. 이 공백이 본 로드맵의 가장 중요한 지표다.

## 5. 현재 작성 현황 및 다음 단계

```

   ├── STD-001 v0.1 ✓ (2026-04-10)
   ├── STD-002 v0.1 ✓ (2026-04-10)
   ├── STD-003 v0.1 ✓ (2026-04-10)
```

### 5.2 다음 단계 — 우선순위 트랙

본 표준 4종이 병행 작성되어 동시에 v0.1로 배포되었으므로, 이후 작업은 **단일 순서가 아니라 독립적으로 병행 가능한 트랙**으로 분류한다.

| 트랙 | 대상 | 근거 | 권고 우선순위 |
|---|---|---|---|
| **트랙 A (운영 병목 해소)** | OPG-001 착수 | 4.4절의 4개 공급 항목이 모두 OPG-001 소관. 이 문서 없이는 본 표준 어느 것도 실제 적용 불가 | **최우선** |
| **트랙 B (근거 확보)** | TR-001 착수 | 가중치 이의 제기 방어. STD-001 미결 항목 해소 | 높음 |
| **트랙 C (정합성 검증)** | 본 표준 4종 v0.1 상호 정합성 감사 | 네 문서가 병행 작성되어 상호 참조가 형성되었으나, 교차 검증은 아직 | 높음 |
| **트랙 D (인증 체계)** | STD-004 착수 | STD-005가 이미 certificationLevel 필드를 참조하며 설계됨 | 중간 |
| **트랙 E (구현 지원)** | 오픈소스 진단 도구 착수 | 7장 참조 | 중간 (트랙 A와 병행 가능) |

**트랙 A와 B는 동시에 진행 가능하며 서로 독립적이다.** 트랙 C는 단독 세션으로 진행하되, 트랙 A·B에서 나오는 결정이 본 표준에 반영되는 창구를 미리 정해 두어야 한다.

- 데이터 유형별 특화 기준 (STRUCT·TEXT·IMAGE·TSERIES·**FT**) (STD-001)

## 7. 리스크 및 병목

### 7.1 운영 단계 진입 병목

본 표준 네 문서가 v0.1로 공식 배포되었음에도 **아직 실제 운영에 적용할 수 없는** 구조적 이유는 다음과 같다.

| 리스크 | 원인 | 완화 방안 |
|---|---|---|
| **가중치 이의 제기에 대응 불가** | TR-001 부재로 D3=0.25 등 수치 근거 미공개 | 트랙 B 착수. 임시로는 STD-001 본문에 "차기 개정에서 근거 제시 예정" 주석 추가 |
| **Q-Tier 판정이 언제까지 유효한지 미정** | OPG-001 부재로 유효기간 없음 | 트랙 A 착수. 임시로는 "1년"을 STD-001 주석에 참고값으로 제시 가능 |
| **어떤 진단 도구의 결과를 받아들일지 미정** | OPG-001 부재로 도구 인증 기준 없음 | 트랙 A 착수. 임시로는 "Self-Attested" 수준만 허용 |
| **certificationLevel 필드가 선언만 되고 연동 없음** | STD-004 미작성 | 트랙 D 착수. STD-005는 제정 전까지 해당 필드를 `null` 허용으로 유지 |
| **본 표준 간 v0.1 상호 참조의 실제 정합성 미검증** | 네 문서가 병행 작성되어 교차 검증 누락 가능 | 트랙 C 정합성 감사 세션 편성 |

### 7.2 정합성 드리프트 리스크

본 표준 v0.1이 각각 독립 개정되는 과정에서 상호 참조 관계가 어긋날 위험이 있다. 특히 다음 세 영역이 민감하다.

1. **purposeReadinessVector** — STD-001과 STD-002 Layer 3가 공동 설계. 한쪽이 개정되면 반드시 동시 개정되어야 한다.
2. **Quality-Ready 전이 조건** — STD-002 7.2절이 단일 출처이며, STD-001 Gate 임계치와 수치 정합성이 유지되어야 한다.
3. **Pack 관련 속성** — STD-002 Layer 3 공통 속성 7개와 STD-003 Pack 구조, STD-005 교환 규격이 세 문서 동시 정합성을 요구한다.

**완화 방안:** 본 표준 개정 시 "상호 참조 영향 분석(Cross-Reference Impact Analysis)"을 개정 체크리스트로 의무화한다. 이 체크리스트는 OPG-001의 규정 대상이 될 수 있다.

## 8. 오픈소스 도구와 표준의 관계

```
AIRD 표준 시리즈 (STD-001/002/003/005 v0.1 공식 배포)
      │
      │ 구현
      ▼
오픈소스 진단·유통 도구 (미착수)
      │
      ├── STD-001 구현: 측정 공식, Q-Tier 판정 알고리즘, JSON 리포트 생성
      ├── STD-002 연계: 리포트 → 3계층 메타데이터 자동 기록
      ├── STD-003 연계: 진단 결과 → 변환 파이프라인 트리거, Gate 판정 자동화
      └── STD-005 연계: Pack 레지스트리 구축, 탐색 API 구현
```

**도구와 표준의 역할 분리 원칙:**

- 표준은 "무엇을"과 "어떻게 판정하는가"를 정의한다
- 도구는 그 정의를 구현한다
- 해석 이견이 발생하면 표준 본문이 우선하며, 도구 소스코드가 표준의 구현 근거가 된다
- 도구가 없어도 표준은 유효하다. 단, 수작업 측정은 STD-001 부속서 A의 즉시 측정 가능 지표(9개) 범위에서 시작 가능하다
